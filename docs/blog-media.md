# ソトバコ ブログメディア 共通ルール

このファイルは、ソトバコが運営するブログメディア全体に適用する共通ルールです。
掲載するサービスが増えても、文体・構成・SEO・画像管理・記事管理のルールはここに一元管理します。

### 参考メディアの使い方

- **記事執筆前に必ず実施**: 同テーマの記事をWebFetchで取得し、「扱っている情報・視点・見出し構成・情報の深さ」を把握する
- **記事内容も参考にする**: 競合が取り上げている情報・悩みの言語化・解説の切り口を参考にして、自社サービス視点で書き直す
- **差別化ポイントを探す**: 競合が書いていない切り口や、自社サービスならではの解決策を盛り込む
- **機能仕様の裏付け**: kintone公式（kintone.cybozu.co.jp / cybozu.dev）で正確な情報を確認する
- **書き方・文体はトヨクモスタイルを優先**: 各社で文体は異なるため、敬体・見出しの付け方・段落構成はトヨクモブログの特徴（下記）に従う

### トヨクモブログから学んだスタイルの特徴

1. **タイトル設計**
   - **フォーマット**: `前半|後半` の形式で区切る
     - 前半：日本語15文字以内（ASCII英数字は文字数に含まない）。読者の悩み・状況を端的に表す
     - 後半：日本語20文字以内。解決策・記事で得られる内容を示す
     - 例：`kintoneのポータルが使いにくい原因｜部署別に整理する具体的方法`
   - メインキーワードを前半に自然に含める
   - 読者が検索するであろう言葉をそのまま含める（「使いにくい」「整理できない」等）

2. **導入文**
   - 読者の悩み・課題に共感するところから始める
   - 「〜に悩んでいませんか？」「〜と感じたことはありませんか？」のような問いかけを使う
   - 記事を読むことで得られるベネフィットを明確にする
   - 200〜300字程度でまとめる

3. **記事冒頭の「この記事でわかること」**
   - 箇条書きで記事で得られる情報を3〜5点提示
   - 読者がスキャンして「自分に有益だ」と判断できるようにする

4. **見出し構成（H2・H3）**
   - H2: 大テーマ（問題提起→解説→解決策→まとめの流れ）
   - H3: 具体的な方法・項目（番号つきが多い：「方法①」「方法②」）
   - 見出しにキーワードを自然に含める
   - 体言止めと文末を使い分ける

5. **本文の書き方**
   - **ですます調**（丁寧語・敬体）を統一して使う
   - 専門用語は初出時に必ず説明する
   - **箇条書きを多用**（3〜5点以内でまとめる）
   - **比較表**（テーブル）を使って情報を整理する
   - 一段落は3〜5文程度に抑え、読みやすく
   - 具体例・シナリオを使って読者がイメージしやすくする

6. **CTA（コールトゥアクション）の配置**
   - 記事の途中（問題の解決策として自社サービスを紹介する箇所）
   - 記事の最後（まとめの後）
   - 押しつけがましくならず、解決策として自然に誘導する

7. **まとめ**
   - 記事の内容を3〜5点で箇条書き
   - メインキーワードを含んだ結論文を書く
   - 次のアクション（フリープランの利用開始や問い合わせ）への誘導で締める

## 記事の基本構成テンプレート

```
【タイトル】
メインキーワード + 読者の悩み/課題 + 解決を示す言葉

【導入文】（200〜300字）
- 読者の悩みへの共感
- 記事で解決できることの予告

【この記事でわかること】
- 箇条書き3〜5点

【H2: 問題提起・現状説明】
- ターゲットが感じている課題を掘り下げる

【H2: 原因の解説】
- なぜその問題が起きるのかを説明

【H2: 解決策〇選】
  【H3: 方法① ○○】
  【H3: 方法② ○○】
  【H3: 方法③ ○○】
  ※自社サービスは最後に持ってきて「一番おすすめ」として紹介

【H2: 自社サービスの詳細紹介】
- 機能・メリット・導入の簡単さ

【H2: こんな企業におすすめ】
- ターゲット像を明示

【まとめ】
- 箇条書きで記事の要点
- CTA（フリープランの利用開始への誘導）
```

### 機能記事の構成テンプレート（⑤⑥⑩）

記事05・06の執筆を通じて確立したパターン。機能記事では「標準機能の紹介 → 限界 → ソトバコポータルで解決」の流れが効果的。

```
【タイトル】
メインキーワード + 標準機能の限界を示す言葉 + 解決策の予告
例: kintoneポータルをダッシュボード化｜グラフと一覧で業務を可視化
例: kintoneお知らせ掲示板の3つの限界｜部署別に情報を届ける方法

【導入文】（200〜300字）
- 読者の悩みへの共感（「毎回アプリを開くのが面倒」等）
- 記事で解決できることの予告

【この記事でわかること】
- 箇条書き3点（標準機能の方法・限界・解決策を予告）

【H2: kintone標準機能で○○する方法】
- 標準機能でできることを具体的な手順で紹介
- 「プラグインなし・無料でここまでできる」と建設的に評価

【H2: 標準機能の3つの限界】
  【H3: 限界①：○○ができない】
  【H3: 限界②：○○ができない】
  【H3: 限界③：○○ができない】
  ※限界は3つに絞る

【H2: ソトバコポータルなら○○が実現できる】
- **比較表（標準機能 vs ソトバコポータル）を必ず設置**
- 限界に対応する解決策を具体的に紹介
  【H3: 機能A の紹介】
  【H3: 機能B の紹介】
- 関連記事（7分記事等）への内部リンク

【まとめ】
- 箇条書き3〜5点
- CTA（フリープラン誘導）
```

## SEO / AIO チェックリスト

記事の執筆・編集時に以下のチェックをすべて通すこと。

### 1. キーワード最適化

- [ ] メインキーワードが以下の4箇所に含まれている: タイトル・最初のH2・導入文・まとめ
- [ ] 関連キーワードが記事全体に自然に分散している（同一KWの過剰繰り返しを避ける）
- [ ] `slug` がメインキーワードの英語/ローマ字でハイフン区切りになっている（例: `kintone-portal-tsukainikunai`）
- [ ] `description`（メタディスクリプション）が120〜130字でキーワードを含んでいる

### 2. 見出し構造

- [ ] H1（タイトル）がページ内に1つだけ存在する
- [ ] 見出し階層が正しい（H2 → H3の順。H2の直下にH4を置かない）
- [ ] 主要な見出し（H2・H3）にメインキーワードまたは関連キーワードが自然に含まれている
- [ ] 見出しだけを読んで記事全体の流れが把握できる（目次として機能する）

### 3. コンテンツ品質

- [ ] 記事種別に応じた文字数を満たしている（共感:3,000〜4,000字 / 方法:4,000〜5,000字 / 機能:3,000字 / 7分:2,500字）
- [ ] 専門用語は初出時に説明している
- [ ] 具体例・数字・比較表など、読者が判断材料にできる情報がある
- [ ] kintoneの機能仕様について、公式ドキュメント（cybozu.dev / kintone.cybozu.co.jp）と整合している
- [ ] 情報の時点が明記されている（「2026年2月時点」等。料金・仕様は変更される可能性がある）

### 4. 内部リンク・回遊

- [ ] 回遊構造（共感→方法→機能→7分→LP）に従ったリンクが設置されている
- [ ] まとめ直前にCTA（LP・フリープランへの誘導）が配置されている
- [ ] 関連する既存記事への内部リンクが1つ以上ある
- [ ] 内部リンクのパスが `/articles/スラッグ` 形式で正しい

### 5. 画像・メディア

- [ ] Alt テキストが日本語でキーワードを含んでいる
- [ ] 画像パスが絶対パス（`/images/...`）になっている
- [ ] 画像配置ルール（H2/H3の体裁統一）に準拠している
- [ ] 新規画像は `images/README.md` に追記されている

### 6. frontmatter

- [ ] `title`: タイトル設計ルール（前半｜後半形式）に準拠
- [ ] `slug`: 英語/ローマ字ハイフン区切り、URLとして有効
- [ ] `description`: 120〜130字、メインキーワード含む
- [ ] `publishedAt`: `YYYY-MM-DD` 形式
- [ ] `category`: 記事種別と一致（`kyoukan` / `houhou` / `kinou` / `7min` / `service`）
- [ ] `tags`: メインキーワード＋関連キーワードを配列で設定
- [ ] `ogImage`: OGP画像パスが設定されている
- [ ] `articleNumber`: 記事一覧のNo.と一致
- [ ] `faq`: 検索されそうな質問を2〜5問設定（FAQ JSON-LD自動生成に使用）

### 7. AIO（AI検索最適化）

AI検索（Google AI Overview、Perplexity等）で引用・参照されるための最適化。

#### 構造化された回答

- [ ] 記事冒頭（導入文 or 「この記事でわかること」）で、想定される検索質問に対する直接的な回答を提示している
- [ ] 手順は番号付きリスト、比較はテーブルなど、AIが構造を認識しやすい形式で記述している
- [ ] 各セクションの冒頭で結論を述べてから詳細を説明している（結論→理由→詳細の順）

#### FAQ最適化

- [ ] `faq` frontmatter に、ユーザーが実際に検索しそうな質問を設定している
- [ ] 各FAQ回答は1〜3文で簡潔にまとまっている（AI抽出に適した長さ）
- [ ] 質問文は「〜とは？」「〜する方法は？」「〜できますか？」など自然な疑問形

#### E-E-A-Tシグナル

- [ ] kintoneパートナーとしての立場に言及している（権威性: Authoritativeness）
- [ ] 実際の操作画面スクリーンショットを使用している（経験: Experience）
- [ ] 情報の時点・出典を明記している（信頼性: Trustworthiness）
- [ ] 公式ドキュメントへの参照がある場合はリンクを記載している

#### エンティティ最適化

- [ ] 固有名詞の表記が正式名称で統一されている（例: 「kintone」は小文字始まり、「ソトバコポータル」はカタカナ）
- [ ] サービスの属性（提供元・カテゴリ・用途）が記事内で明記されている

#### 文体バランス

- [ ] 箇条書き・テーブル・番号リストを活用し、テキストの壁を避けている
- [ ] 一段落は3〜5文以内に収まっている
- [ ] 「ですます調」で統一されている

## 画像管理ルール

### 記事内の画像配置ルール

記事内でH2・H3に画像を入れる場合は**体裁を統一**すること。

- **H2の画像**: H2直下に画像を入れるなら、同じ記事内のすべてのH2（「この記事でわかること」「まとめ」を除く）に画像を入れる。入れないなら全H2に入れない。
- **H3の画像**: あるH2セクション内のH3群で画像を入れるなら、そのH3群すべてに画像を入れる。入れないなら全H3に入れない。
- **該当画像がない場合**: `images/no_image.jpg` をプレースホルダーとして使用する。ユーザーが後から実画像に差し替える。
- **「まとめ」直下には画像を入れない。**

### README.mdの「使いどころ」記載ルール

画像を記事に配置した際は、`images/README.md` の「使いどころ」欄に、差し込んだ箇所の周辺文章を読み込んだうえで、どの記事・どのセクション・何を説明する文脈で使われるかを具体的に記載する。

### フォルダ・カタログ

- 画像はすべて `images/` フォルダに格納する
- 画像の詳細情報は `images/README.md`（画像カタログ）で一元管理する
- **新しい画像を追加したら必ずカタログに追記する**

### 命名ルール

```
{feature}_{description}.png

例: announce_category.png
```

- `feature`: 機能名（announce / tab / graph / app-list / notification / unprocessed / space / smartphone / setup / kintone / portal）
- `description`: 何の画面か（overview / settings / before / after / category / list / drag-drop / mobile / add）
- すべて小文字・ハイフン区切り・拡張子は `.png`（写真素材は `.jpg`）

### 記事への埋め込み方法

```markdown
![Altテキスト](../images/ファイル名.png)

例:
![ソトバコポータルの全体アナウンス・カテゴリ別表示画面](../images/announce_category.png)
```

- **Altテキストは `images/README.md` のカタログからコピーする**（SEO効果あり）
- 画像選定に迷ったら `images/README.md` の「使いどころ」列を参照する

### 画像追加フロー

```
1. スクリーンショット・素材を用意する
2. images/ フォルダ直下に置く（GitHub Web UIまたはgit push）
3. Claudeに「画像を追加した」と伝える（他社サイトからの引用の場合は出典URLも伝える）
4. Claudeが画像を確認し、命名規則に従ったリネーム・該当サブフォルダへの移動・images/README.md への追記を行う
5. 他社のサイト・外部素材からの引用画像は images/README.md の「引用画像の出典一覧」にも追記する
```

### 引用画像の出典記載ルール

- **他社のWebサイト・ヘルプページ・外部素材から画像を引用した場合**は、`images/README.md` の「引用画像の出典一覧」に必ず出典を記載する
- 自社で撮影・作成した画像（サービスを自社環境で操作したスクリーンショット等）は記載不要
- **記事内で引用画像を使用する際**は、画像の直下に出典表記を追加する：
  ```markdown
  出典：[サイト名](URL)
  ```
- 引用前に出典元の利用規約・著作権を確認すること

## 記事制作方針

### 選定基準

- **担当者の痛みが強い** × **契約に近い** × **競合が薄い** → 短期回収グループ（①〜④）
- **機能検索・比較検討層** × **上位プラン検討者** × **ロングテール回収** → 検討・機能グループ（⑤〜⑩）

### 執筆時の鉄則

- **ノウハウを全部書かない**: 記事で「何が必要か」は伝えるが、「どうやるか」の細部はサービスの操作に委ねる
- **ただの解説記事にしない**: 読んで終わりではなく、次のアクション（LP・関連記事）に動線をつなぐ
- **ページ単体で完結させない**: 内部リンクで回遊させ、滞在時間と接触回数を増やす
- **必ず「最短の導線」を作る**: まとめ直前に「管理者1人でやるなら、この方法が最短です」という一文でCTAへ誘導する
- **詳しい外部記事はおすすめしてOK**: 標準機能の手順など、他サイトにわかりやすい記事がある場合はリンクで紹介して構わない。自社記事で全てを書き切る必要はなく、読者にとって有益な情報源を案内する姿勢が信頼につながる

### 内部リンクによる回遊構造

```
共感記事（①④⑨）    3,000〜4,000字
      ↓ 内部リンク
方法記事（②③⑧）    4,000〜5,000字
      ↓ 内部リンク
機能記事（⑤⑥⑩）   3,000字程度
      ↓ 内部リンク
7分記事（⑦）        2,500字程度
      ↓ CTA
LP（https://sotobaco.com/sotobacoportal）
```

- **共感記事 → 方法記事**: 「なぜ起きるのか」に共感させた後、「どう解決するか」の記事へ誘導
- **方法記事 → 機能記事**: 「整理したい」意欲が高まった状態で、具体機能の深掘り記事へ誘導
- **機能記事 → 7分記事**: 機能に興味を持ったユーザーを「すぐ試せる」手順記事へ誘導
- **7分記事 → LP**: 導入イメージができた状態でLPに送り、フリープラン登録へ誘導
