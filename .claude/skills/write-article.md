# /write-article — SEO記事執筆スキル

## 概要
kintone関連市場向けのSEO記事を、調査から執筆・仕上げまで一貫して行う。

## 実行方針
- **途中確認不要**: 調査→構成→執筆→完了報告まで、許可を求めず一気通貫で進める
- Bashコマンド（npm run dev, open, git操作など）もすべて許可済みとして実行する
- MCPツール（YouTube Transcript, WebFetch, WebSearch等）もすべて許可済みとして実行する
- 構成案の確認も不要。完成した記事をブラウザで表示し、完了報告のみ行う
- **Playwrightは使用禁止**: 記事執筆フローでPlaywright MCPを呼び出さない。ローカル確認は `open` コマンドのみ

## 実行手順

### 1. 事前準備
- `docs/blog-media.md` を読み込む（記事制作方針・スタイル・回遊構造・記事一覧）
- `docs/company.md` を読み込む（共通注意事項・URL）
- 対象サービスの docs を読み込む（`docs/sotobaco-portal.md` / `docs/btone.md` 等）

### 2. キーワード調査（必須）
- テーマに関連する検索キーワードをWebSearchで複数パターン調査する
- kintone標準の機能名・用語を確認する（ユーザーが実際に検索する用語と、社内で使われる用語は異なることが多い）
- 競合記事が使っているキーワードを確認し、検索ボリュームが見込める用語を特定する
- 検索されない独自用語をメインキーワードにしない（例: 「全社アナウンス」→「お知らせ掲示板」）
- 調査結果をもとにメインキーワード・関連キーワードの候補を確定する

### 3. 競合・参考メディア調査
- `docs/blog-media.md` の参考メディア一覧から、同テーマの記事をWebFetchで取得
- 扱っている情報・視点・見出し構成・情報の深さを把握する
- **YouTube動画も調査する**: WebSearchで関連動画を探し、YouTube Transcript MCP（`get_transcript`, `lang: "ja"`）で上位3〜5本の文字起こしを取得。動画でしか語られていないユーザーの悩み・実践ノウハウを記事に活かす
- 差別化ポイントを特定する（競合が書いていない切り口、自社ならではの解決策）
- kintone公式（kintone.cybozu.co.jp / cybozu.dev）で機能仕様の裏付けを取る

### 4. 構成設計
- 記事種別を確認（共感 / 方法 / 機能 / 7分）
- `docs/blog-media.md` の基本構成テンプレートに従って見出し構成を作成
- ステップ2で確定したメインキーワード・関連キーワードを使用する
- 文字数目安: 共感3,000〜4,000字 / 方法4,000〜5,000字 / 機能3,000字 / 7分2,500字
- 構成案は自分で判断して確定する（途中確認不要）

### 5. 執筆
- トヨクモブログスタイルに従う（ですます調、箇条書き多用、比較表活用）
- タイトル: `前半|後半` 形式（前半15文字以内 / 後半20文字以内）
- 導入文: 読者の悩みに共感 → ベネフィット提示（200〜300字）
- 「この記事でわかること」を箇条書き3〜5点で設置
- 自社サービスは解決策の最後に「一番おすすめ」として紹介
- CTA: 記事中盤（解決策として自然に）+ 記事末尾（まとめ後）
- まとめ: 箇条書き3〜5点 + CTA（フリープラン誘導）

### 6. SEO・内部リンク
- メインキーワード: タイトル・最初のH2・導入文・まとめに含める
- 関連キーワード: 自然に散りばめる
- メタディスクリプション: 120〜130字
- 回遊構造に従い内部リンクを設置（共感→方法→機能→7分→LP）

### 7. 画像配置
- H2/H3の画像配置ルールに従う（入れるなら全部、入れないなら全部入れない）
- 「まとめ」直下には画像を入れない
- Altテキストは `images/README.md` からコピー
- 画像がない場合は `images/no_image.jpg` をプレースホルダーとして使用

### 8. 仕上げ
- `docs/blog-media.md` の記事一覧に追記する
- `npm run dev` でdevサーバーを起動し、`open http://localhost:3000/articles/{slug}` でブラウザに該当記事を自動表示する
- 社長に完成報告（記事の要点・文字数・CTA配置箇所）
- **Playwrightチェックは不要**（デザイン確定済みのため）
- **社長の確認が完了するまで push しない**（コミットまでに留め、社長のOKが出てから push → ステージングデプロイ）

## 鉄則
- ノウハウを全部書かない（「何が必要か」は伝え、「どうやるか」の細部はサービスに委ねる）
- ただの解説記事にしない（次のアクションへの動線を必ずつなぐ）
- ページ単体で完結させない（内部リンクで回遊させる）
- 必ず「最短の導線」を作る（まとめ直前に一文でCTAへ誘導）
- 価格はすべて税抜表示
- 詳しい外部記事はおすすめしてOK（標準機能の手順など、他サイトにわかりやすい記事があればリンクで紹介する）

## キーワード選定の注意点（実績から得た知見）

- **ソトバコポータルの独自機能名はキーワードにしない**: 「全体アナウンス」「スペースリンク」等はソトバコポータル固有の名前であり、ユーザーが検索する語ではない
- **kintone標準の機能名・一般名称を使う**: 「お知らせ掲示板」「ダッシュボード」「ポータル カスタマイズ」等、ユーザーが実際に検索する語をメインキーワードにする
- **具体例**:
  - NG: 「全体アナウンス」→ OK: 「お知らせ掲示板」（記事06で変更）
  - OK: 「ダッシュボード」（一般用語として検索される。記事05で採用）
  - OK: 「アプリ 増えすぎ 整理」（ユーザーの悩みをそのままキーワードに。記事03で採用）

## 記事種別ごとの執筆パターン

### 機能記事パターン（⑤⑥⑩）

「標準機能紹介 → 3つの限界 → ソトバコポータルで解決」の流れが効果的。

**構成テンプレート**:
1. 導入文（読者の悩みに共感）
2. kintone標準機能でできること（手順をシンプルに紹介）
3. 標準機能の3つの限界（具体的な課題を番号付きで）
4. ソトバコポータルの解決策（**比較表** + 機能紹介）
5. まとめ + CTA

**ポイント**:
- 標準機能を「ダメ」と否定しない。「ここまではできるが、ここから先は限界」と建設的に書く
- 限界は**3つに絞る**（多すぎると散漫、少なすぎると説得力不足）
- **比較表（標準機能 vs ソトバコポータル）を必ず入れる**（読者が一目で違いを把握できる）
- 各限界に対するソトバコポータルの解決策を1対1で対応させる
- 標準機能の仕様・限界の根拠は `docs/sotobaco-portal.md` の「kintone標準機能の仕様と限界」セクションを参照する

**競合との差別化**:
- 競合記事はプラグイン紹介（krewDashboard, kViewer等）が多く、標準機能の深掘り→限界→解決策の流れが薄い
- 「標準機能でどこまでできるか」を丁寧に解説することが差別化ポイント

### 方法記事パターン（②③⑧）

「悩みの共感 → 原因の構造化 → 複数の解決策 → ソトバコポータルを最推奨」の流れ。

**ポイント**:
- 解決策は3〜5個提示し、ソトバコポータルは最後に「一番おすすめ」として紹介
- 各解決策のメリット・デメリットを正直に書く（デメリットを書くことで信頼性が上がる）
- 「自社のアプリ数・部署数に応じた選び方」など、判断基準を示すと親切

### 共感記事パターン（①④⑨）

「あるある共感 → 原因分析 → 解決の方向性提示 → 方法記事・機能記事へ誘導」の流れ。

**ポイント**:
- 導入文で「あなたもこう感じていませんか？」と読者の言葉で悩みを代弁する
- 原因を構造的に分析することで「なるほど、だからうまくいかないのか」と腹落ちさせる
- 記事内では解決の方向性だけ示し、具体的な手法は方法記事・機能記事にリンクで誘導する
