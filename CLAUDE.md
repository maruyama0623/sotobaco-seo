### 主な機能
- **タブ整理・閲覧制限**: アプリをタブごとに分類し、ユーザー・グループ・組織別に閲覧権限を設定
- **視認性カスタマイズ**: 色分け（7色から選択）・アイコン設定でアプリを直感的に識別
- **ドラッグ&ドロップ**: 簡単にレイアウト変更
- **アプリカード種類**: 3種類から選択して追加できる
  - **kintoneアプリリンク**: kintoneアプリへのリンクカード（アプリアイコン自動取得）
  - **外部リンク**: スプレッドシートや勤怠システムなど外部サービスのURLを登録
  - **スペースリンク**（スタンダード）: kintoneスペースへのリンクカード。スペース名・カバー画像はkintone側の変更を自動同期
- **アプリカード設定項目**: サイズ（大/小）・リンクの開き方（別タブ/同じタブ）・カラー（7色）・説明文・アイコンを設定可能。サイズ「大」は説明文が展開表示され、「小」はコンパクト表示
- **グラフリアルタイム表示**: kintoneのグラフをポータル上にリアルタイム表示（サイズS/L選択可）
- **アプリ一覧埋め込み**: kintoneのアプリ一覧をポータルに表示
- **通知一覧表示**: kintoneの通知をポータルに表示（絞り込みは元の通知一覧から操作）
- **未処理一覧表示**: kintoneの未処理レコードをポータルに表示
- **全体アナウンス機能**: 1〜10件をポータルに表示（デフォルト3件）、カテゴリ分け対応（スタンダード）
- **限定公開機能**: 全社導入前に一部ユーザー・グループでテスト適用
- **アプリカード位置固定**: 画面サイズが変わっても迷わない
- **スマートフォン対応**: JSファイルをkintone全体カスタマイズにアップロードすることでモバイル対応
- **スペース単位ポータル**: スペースIDを設定してスペースごとに異なるポータルを表示（スタンダード）
- **kintoneデフォルトポータル非表示**: 対象ユーザー・グループ・組織ごとにkintone標準ポータルを非表示化

### 導入手順（最短3ステップ）

1. **アカウント作成** → ソトバコポータルにアクセスしてアカウントを作成
2. **kintone連携設定** → cybozu.com共通管理画面でOAuthクライアントを追加し、クライアントID・シークレットをソトバコポータルに入力して接続（共通管理者権限が必要）
3. **ポータル設定** → タブ・グループ・アプリカードを追加して「ポータルを更新する」をクリック

> 「最短7分」での導入が可能。kintone連携はOAuth認証を使用するため、cybozu.com共通管理者権限を持つユーザーが設定を行う必要がある。

### 操作ガイド補足情報（記事執筆時に活用）

- **ポータル反映**: 設定変更後は必ず「ポータルを更新する」ボタンをクリックしないとkintoneに反映されない
- **スペースID**: kintoneのスペースURLの末尾の数字（`/k/#/space/〇〇〇` の〇〇〇部分）
- **スペースリンクの自動同期**: kintone側でスペース名・カバー画像を変更すると、ポータルのスペースリンクカードにも自動で反映される（手動更新不要）
- **全体アナウンスカテゴリ**: kintoneのフィールドコードを必ず `category` にする必要がある
- **スマートフォン対応**: `portal-design-common.min.js` をkintone全体カスタマイズのスマートフォン用JSとしてアップロード
- **ポータルが表示されない場合の主な原因**:
  - kintoneシステムのアクセス権（アプリの管理）が付与されていない
  - ポータルデザイン設定反映アプリの権限（レコード追加・アプリ管理・ファイル読み込み）が不足
  - カスタマイズ適用範囲が「すべてのユーザーに適用」になっていない
  - ブラウザ・iOSのバージョンが古い

### 参考メディアの使い方

- **記事執筆前に必ず実施**: 同テーマの記事をWebFetchで取得し、「扱っている情報・視点・見出し構成・情報の深さ」を把握する
- **記事内容も参考にする**: 競合が取り上げている情報・悩みの言語化・解説の切り口を参考にして、ソトバコポータル視点で書き直す
- **差別化ポイントを探す**: 競合が書いていない切り口や、ソトバコポータルならではの解決策を盛り込む
- **機能仕様の裏付け**: kintone公式（kintone.cybozu.co.jp / cybozu.dev）で正確な情報を確認する
- **書き方・文体はトヨクモスタイルを優先**: 各社で文体は異なるため、敬体・見出しの付け方・段落構成はトヨクモブログの特徴（下記）に従う

### トヨクモブログから学んだスタイルの特徴

1. **タイトル設計**
   - メインキーワードをタイトルに自然に含める
   - 「〇選」「徹底解説」「方法を解説」など記事の具体性を示す表現を使う
   - 読者が検索するであろう言葉をそのまま含める（「使いにくい」「整理できない」等）

2. **導入文**
   - 読者の悩み・課題に共感するところから始める
   - 「〜に悩んでいませんか？」「〜と感じたことはありませんか？」のような問いかけを使う
   - 記事を読むことで得られるベネフィットを明確にする
   - 200〜300字程度でまとめる

3. **記事冒頭の「この記事でわかること」**
   - 箇条書きで記事で得られる情報を3〜5点提示
   - 読者がスキャンして「自分に有益だ」と判断できるようにする

4. **見出し構成（H2・H3）**
   - H2: 大テーマ（問題提起→解説→解決策→まとめの流れ）
   - H3: 具体的な方法・項目（番号つきが多い：「方法①」「方法②」）
   - 見出しにキーワードを自然に含める
   - 体言止めと文末を使い分ける

5. **本文の書き方**
   - **ですます調**（丁寧語・敬体）を統一して使う
   - 専門用語は初出時に必ず説明する
   - **箇条書きを多用**（3〜5点以内でまとめる）
   - **比較表**（テーブル）を使って情報を整理する
   - 一段落は3〜5文程度に抑え、読みやすく
   - 具体例・シナリオを使って読者がイメージしやすくする

6. **CTA（コールトゥアクション）の配置**
   - 記事の途中（問題の解決策として自社サービスを紹介する箇所）
   - 記事の最後（まとめの後）
   - 押しつけがましくならず、解決策として自然に誘導する

7. **まとめ**
   - 記事の内容を3〜5点で箇条書き
   - メインキーワードを含んだ結論文を書く
   - 次のアクション（フリープランの利用開始や問い合わせ）への誘導で締める

## 記事制作方針（全10記事・半年計画）

### 選定基準

- **担当者の痛みが強い** × **契約に近い** × **競合が薄い** → 短期回収グループ（①〜④）
- **機能検索・比較検討層** × **上位プラン検討者** × **ロングテール回収** → 検討・機能グループ（⑤〜⑩）

### 記事一覧（制作順）

| No | タイトル | 種別 | 文字数目安 |
|----|---------|------|----------|
| ① | kintoneのポータルが使いにくい原因と、部署別に整理する具体的方法 | 共感記事 | 3,000〜4,000字 |
| ② | kintoneで部署ごとにアプリを見せ分ける方法【管理者向け】 | 方法記事 | 4,000〜5,000字 |
| ③ | kintoneのアプリが増えすぎたときの整理術【30個以上ある企業向け】 | 方法記事 | 4,000〜5,000字 |
| ④ | kintone管理者がつらいと感じる3つの理由と解決策 | 共感記事 | 3,000〜4,000字 |
| ⑤ | kintoneポータルをダッシュボード化する方法（一覧・グラフ活用） | 機能記事 | 3,000字程度 |
| ⑥ | kintoneで全社アナウンスを効率的に表示する方法 | 機能記事 | 3,000字程度 |
| ⑦ | kintoneのポータルを7分でカスタマイズする手順 | 7分記事 | 2,500字程度 |
| ⑧ | kintoneのスペース管理がうまくいかない理由と改善策 | 方法記事 | 4,000〜5,000字 |
| ⑨ | kintoneのポータルを整理しないと起きる5つの問題 | 共感記事 | 3,000〜4,000字 |
| ⑩ | kintoneで"今日やること"をポータルに表示する方法 | 機能記事 | 3,000字程度 |

### 執筆時の鉄則

- **ノウハウを全部書かない**: 記事で「何が必要か」は伝えるが、「どうやるか」の細部はソトバコポータルの操作に委ねる
- **ただの解説記事にしない**: 読んで終わりではなく、次のアクション（LP・関連記事）に動線をつなぐ
- **ページ単体で完結させない**: 内部リンクで回遊させ、滞在時間と接触回数を増やす
- **必ず「最短の導線」を作る**: まとめ直前に「管理者1人でやるなら、この方法が最短です」という一文でCTAへ誘導する

### 内部リンクによる回遊構造

```
共感記事（①④⑨）    3,000〜4,000字
      ↓ 内部リンク
方法記事（②③⑧）    4,000〜5,000字
      ↓ 内部リンク
機能記事（⑤⑥⑩）   3,000字程度
      ↓ 内部リンク
7分記事（⑦）        2,500字程度
      ↓ CTA
LP（https://sotobaco.com/sotobacoportal）
```

#### リンク設計の考え方

- **共感記事 → 方法記事**: 「なぜ起きるのか」に共感させた後、「どう解決するか」の記事へ誘導
- **方法記事 → 機能記事**: 「整理したい」意欲が高まった状態で、具体機能の深掘り記事へ誘導
- **機能記事 → 7分記事**: 機能に興味を持ったユーザーを「すぐ試せる」手順記事へ誘導
- **7分記事 → LP**: 導入イメージができた状態でLPに送り、フリープラン登録へ誘導

## SEOの基本ルール

- **メインキーワード**: タイトル・最初のH2・導入文・まとめに含める
- **関連キーワード**: 記事全体に自然に散りばめる（詰め込みすぎない）
- **文字数**: 種別によって異なる（共感:3,000〜4,000字 / 方法:4,000〜5,000字 / 機能:3,000字 / 7分:2,500字）
- **内部リンク**: 回遊構造に従い、各記事に必ず次のステップへのリンクを設置する
- **画像**: 図解やスクリーンショットを使う（Altテキストに日本語のキーワードを設定）
- **メタディスクリプション**: 120〜130字でキーワードを含めた説明文を用意

## 画像管理ルール

### フォルダ・カタログ

- 画像はすべて `images/` フォルダに格納する
- 画像の詳細情報は `images/README.md`（画像カタログ）で一元管理する
- **新しい画像を追加したら必ずカタログに追記する**

### 命名ルール

```
{feature}_{description}.png

例: announce_category.png
```

- `feature`: 機能名（announce / tab / graph / app-list / notification / unprocessed / space / smartphone / setup / kintone / portal）
- `description`: 何の画面か（overview / settings / before / after / category / list / drag-drop / mobile / add）
- すべて小文字・ハイフン区切り・拡張子は `.png`（写真素材は `.jpg`）

### 記事への埋め込み方法

```markdown
![Altテキスト](../images/ファイル名.png)

例:
![ソトバコポータルの全体アナウンス・カテゴリ別表示画面](../images/announce_category.png)
```

- **Altテキストは `images/README.md` のカタログからコピーする**（SEO効果あり）
- 画像選定に迷ったら `images/README.md` の「使いどころ」列を参照する

### 画像追加フロー

```
1. スクリーンショットを撮る
2. 命名ルールに従いファイル名をつけて images/ にアップロード（GitHub Web UIまたはgit push）
3. Claudeに「画像を追加した」と伝える
4. Claudeが画像を確認して images/README.md に追記する
```
